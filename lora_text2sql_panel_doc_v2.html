LoRA: Low-Rank Adaptation,
reproduced & shipped as a Text-to-SQL assistant

A hands-on exploration of Low-Rank Adaptation (LoRA) — a technique from the 2021 research paper by Hu et al. (arXiv:2106.09685, ICLR 2022) that lets us fine-tune a large AI model by training only a tiny fraction of its parameters, while keeping everything else frozen.

What this project is about

The research paper

Hu et al. (2021) noticed that when a large model adapts to a new task, the change in weights has low intrinsic rank. Instead of updating every weight, LoRA freezes the model and learns a small update as the product of two thin matrices: ΔW = (α/r) · B · A. This reduces trainable parameters by up to 10,000× while matching full fine-tuning quality.

One notebook, three stages

  • Verify: We use the industry-standard PEFT library and empirically verify the paper's mechanism inside it — the injected A/B matrices, their initialization, and the frozen base.
  • Fine-tune: We use QLoRA to fine-tune Qwen2.5-0.5B on a Text-to-SQL dataset, training only ~0.4% of its parameters.
  • App: A live interface comparing the base model vs. our fine-tuned LoRA assistant side-by-side.
02 · The paper, in essence

Fine-tune the change, not the model

The main idea of LoRA is simple: instead of changing all the weights of a pretrained model, keep the original weights frozen and add a small trainable module — the LoRA adapter — that learns the new task on top.

Think of it like two parallel paths through the same layer. The frozen path gives the model's original pretrained behavior. The adapter path learns the task-specific change. Both outputs are simply added together.

W′  =  W₀  +  (α / r) · B · A      where  B ∈ ℝd×r,  A ∈ ℝr×k,  r ≪ min(d, k)

Why it works

  • Far fewer parameters to train. A normal 896×896 weight matrix has 802,816 parameters. With LoRA at rank 16, matrices A and B together have only 28,672 — about 3.6% of the original. On GPT-3 175B, the paper reports up to 10,000× fewer trainable parameters with equal or better quality.
  • Memory saving from the optimizer. Adam stores two extra states per trainable parameter (momentum and variance). Freeze 99%+ of weights and those states simply don't exist for the frozen part.
  • Zero inference slowdown. After training, B·A can be permanently merged into W₀ — the layer becomes an ordinary dense matrix again with no extra computation at all.

The initialization trick

  • B = 0 so B·A = 0 at step 0: training starts exactly at the pretrained model. No pretrained knowledge is damaged before training even begins.
  • A ~ Gaussian so gradients can flow immediately. If both A and B were zero, neither could ever learn (gradients would be zero forever). If both were random, BA ≠ 0 at init and the model starts from a damaged state.
  • Scaling α/r keeps the update magnitude stable when the rank changes, so we don't need to re-tune other hyperparameters. Convention: α = 2r (so at r=16, α=32, giving effective scale = 2).

The paper's experiments found that adapting the attention query and value projections (Wq, Wv) gives most of the benefit, and that r = 1–8 already saturates attention adaptation — direct evidence for the low intrinsic rank hypothesis.

04 · System architecture

One frozen base, two personalities

The whole system loads the base model once. The adapter stays unmerged at inference so it can be toggled on and off — one set of weights in GPU memory produces both the "before" and the "after".

Amber dashed line = the adapter artifact traveling from training into the app. It is the only thing that training produces — a few megabytes that turn a generalist model into a SQL specialist.

05 · Reproduction — Mechanism verification

Opening up LoRA inside the PEFT library

We use the industry-standard implementation — Hugging Face PEFT, the same library used in production — and reproduce the paper by verifying its mechanism empirically inside the library. One call to get_peft_model() injects the LoRA layers; we then open them up and check that every claim of the paper actually holds.

What PEFT injects into each attention layer

After injection, each targeted nn.Linear becomes a lora.Linear containing the original frozen layer plus a small trainable bypass. Its components are exactly the ones described in the paper:

ComponentShapeInitWhat it does
W₀, bias frozenout × inoriginal pretrained layer, wrappedThe original frozen weights, kept in place. requires_grad=False — the optimizer never sees them.
Matrix A trainabler × inKaiming / GaussianDown-projection: compresses the input from full size down to r dimensions (e.g. 896 → 16). Random init so gradients can flow from the start.
Matrix B trainableout × rzerosUp-projection: expands back to full output size (e.g. 16 → 896). Zero init ensures ΔW = 0 at step 0, so training starts exactly at the pretrained model.
scalingscalarα / rKeeps the update magnitude stable across different ranks. Convention: α = 2r (so at r=16, α=32, scale=2).
dropoutp = 0.05—Applied only to the LoRA path input. Prevents the small adapter from overfitting on limited training data.

The forward pass

out = F.linear(x, W₀, bias) # frozen path — pretrained behavior, untouched + scaling · ((dropout(x) @ Aᵀ) @ Bᵀ) # trainable bypass — the entire innovation

Four checks — each reproduces a specific paper claim

1 · Injection audit

Print the model's module tree before and after get_peft_model(). Before: q_proj is a plain Linear(896, 896). After: it contains the frozen original plus lora_A (896→16) and lora_B (16→896).

↳ paper claim: LoRA adds a low-rank bypass, base weights untouched

2 · Initialization & gradients

Pull the actual tensors out of the injected layer and assert B == 0 everywhere while A is random. Then confirm only lora_A/lora_B have requires_grad=True, and print the trainable count (~0.4%).

↳ paper claims: ΔW = 0 at init · tiny trainable fraction

3 · Parity at init

Before any training, generate with the adapter attached and again inside with model.disable_adapter(): — the outputs must be identical, proving the untrained adapter changes nothing.

↳ paper claim: stable start exactly at the pretrained model

4 · Lossless merge

After training, merge_and_unload() folds (α/r)·B·A into W₀ — the layer becomes a plain Linear again, and its output matches the unmerged model exactly.

↳ paper claim: zero added inference latency

Positioning for the viva: "I used PEFT — the production standard — and verified every mechanism of the paper empirically inside it, then measured the improvement it buys on a real task." Not hiding the library; demonstrating mastery through it.
06 · Proof of Concept

Fine-tuning a real model for Text-to-SQL

In the same notebook, we attach LoRA adapters to the model's attention layers and train only those — using QLoRA (vanilla LoRA adapters on a 4-bit-quantized frozen base) on a free Colab T4 GPU. The whole project runs end-to-end in one Colab notebook: verify → fine-tune → evaluate → app.

What we use and why

ChoiceWhatWhy
Base modelQwen2.5-0.5B-InstructSmall enough to train on a free T4 GPU (~4 GB VRAM), but powerful enough to produce meaningful SQL. Trains in ~30–45 min.
Datasetb-mc2/sql-create-context78,577 examples of (question, CREATE TABLE schema, SQL). Providing the schema as context stops the model from hallucinating table/column names. We use 3–5k for training and ~200 held-out for evaluation.
QuantizationQLoRA · 4-bit NF4The frozen base model is stored in 4-bit precision (load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_use_double_quant=True), cutting GPU memory by ~75%. Computations run in fp16. The LoRA adapter stays in 16-bit because it is the part being trained.
Target layersq_proj, k_proj, v_proj, o_projLoRA adapters are attached to all four attention projections. The paper found q+v carry the most task signal; adding k and o gives a small further improvement at negligible cost on a 0.5B model.

Key training parameters

ParameterValueReason

Prompt template

You are a text-to-SQL model. Given a database schema and a question, output only the SQL query. ### Schema: {CREATE TABLE ...} ### Question: {natural language question} ### SQL: {target — loss computed here only}

07 · Evaluation

Comparing base vs LoRA — honestly

Both models see identical prompts. The comparison runs through one loaded model: wrap generation in disable_adapter() for the base output, call normally for the adapted output.

-- Q: "How many heads of departments are older than 56?" -- Schema: CREATE TABLE head (age INTEGER) Sure! To find the number of heads older than 56, you could write a query like: SELECT * FROM heads WHERE the age is greater than 56... (invented table name, chatty, invalid)
Typical zero-shot behavior: conversational filler, hallucinated table names, occasionally invalid syntax.

Metrics, in order of importance

  1. Execution accuracy (primary). For each held-out example (~100–200): build the table in an in-memory SQLite DB from its CREATE statement, populate a few synthetic rows, execute gold SQL and generated SQL, compare result sets. Credits semantically-correct variants that exact-match would wrongly fail (age > 25 vs 25 < age).
  2. Exact / normalized match (secondary). Reported with its known weakness stated.
  3. Qualitative before/after. 8–12 curated examples in slides and the live app — the most persuasive artifact for a panel.

Figures & tables produced

  • Trainable-params comparison: full FT vs LoRA (count + %) — from print_trainable_parameters()
  • Training loss curve · Base-vs-LoRA accuracy bar chart · Peak VRAM & wall-time table
  • Rank ablation r = 4 / 8 / 16: accuracy vs adapter size (time permitting — reproduces a core paper finding)
Stated limitation: CREATE statements carry no data rows, so execution accuracy uses synthetic row population — noted transparently in the report, with schema-only comparison as fallback for aggregate queries.
08 · The application

A working SQL assistant, not just a notebook

We build a Gradio web app where users type a database schema and a question in plain English — and instantly see the SQL generated by both the original base model and the LoRA fine-tuned model, side by side.

How it works

  • User enters a question in English and a CREATE TABLE schema
  • One Generate SQL button → fills two side-by-side gr.Code panels: Base model output vs LoRA fine-tuned output
  • The base model is loaded only once. The adapter is toggled on or off using disable_adapter() — two different behaviors from one set of weights in GPU memory
  • Run on sample DB button — executes both SQL queries on an in-memory SQLite database and shows the result tables live
  • Pre-loaded example queries for a smooth demo flow

Shared via launch(share=True) from Colab, with a pre-recorded video as backup.

What the comparison shows

  • Base model: often produces incorrect SQL, invents column or table names, or gives a chatty explanation instead of a clean query
  • LoRA fine-tuned model: produces clean, schema-grounded SQL using the exact column and table names from the schema
  • The improvement comes entirely from a few-MB adapter — the original 494M-parameter base model is completely unchanged
  • The unmerged adapter adds tiny latency by design (needed for the toggle comparison). Calling merge() removes even this overhead in production — as proven by the merge test in §05
